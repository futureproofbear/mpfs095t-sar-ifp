# sar_app -- SAR application as an HSS payload (Discovery Kit, MPFS095T)

Autonomous microSD boot of the SAR focus pipeline. HSS (in eNVM) trains DDR,
reads the GPT microSD, loads this payload to DDR and jumps to it; the app then
runs the whole pipeline with no JTAG/gdb/mailbox in the loop.

## Build (board-free)
```bash
./build_sar_app.sh          # -> Default/sar_app.elf + Default/sar_app.bin
```
Env mirrors `../hss/build_hss.sh` (SoftConsole v2022.2 riscv64-unknown-elf gcc
8.3.0, rv64gc/lp64d/medany). Compiles the mpfs-hal + the SAR `src/` tree against
the disco-kit MSS design config
(`src/boards/mpfs-discovery-kit/fpga_design_config`, generated by the HSS build)
and the payload MSS software config in `config/`.

## What the payload does (`u54_1.c`, `#ifdef SAR_SD_BOOT`)
```
init MMUART_0 console
FFTMODE=1 (0xB0059110)  -- FABRIC CoreFFT, written UNCONDITIONALLY (no CPU fallback)
sar_sd_init()           -- microSD: card_type SD, 4-bit, 3.3 V, NO Icicle mux
sar_sd_load_scene(0)    -- 'SARI' superblock at SAR_SD_SCENE_LBA (GPT P2 = 67584)
sar_form_image(0)       -- full PFA pipeline -> SAR_SEQ_OK
sar_sd_save_out(...)    -- persist OUT to SD (commit-last SAVEOUT)
```

## Key parameters
- DDR exec address: `0x1000000000` (`sar_app.ld`, `sar_app.yaml`, ELF entry). On
  the disco SEG config this aliases the 32-bit window `0x80000000`; app code
  occupies DDR physical [0, 128 MiB), clear of the SAR SIG buffer at `0x88000000`
  (physical 128 MiB). The fabric FIC0 masters keep driving the fixed 32-bit
  buffer addresses (`sar_kernels.h` / `ddr_sar_layout.h`) -- unchanged.
- `SAR_SD_SCENE_LBA = 67584` (`src/sar/sar_sd.h`) -- GPT P2 base, pinned by
  `mpfs/host/sd_pack.py --gpt`. TOC entries carry absolute card LBAs.
- `SAR_SD_OUT_LBA` -- output-persist base; OPEN board-side item (needs a reserved
  P3 output region pinned in `sd_pack.py`, parallel to `SAR_SD_SCENE_LBA`).

## Payload packaging (needs a native host gcc -- not on this PC)
```bash
hss-payload-generator -c sar_app.yaml payload.bin   # from hart-software-services/tools
```
Then `sd_pack.py --gpt` lays P1 = payload.bin, P2 = 'SARI' scene at LBA 67584.

## Board validation still required (NOT proven here)
Clean build only. Runtime unproven: SD enumeration + speed, the exec-addr/SEG
alias, and the FIC0 flush pairing all need on-Discovery bring-up. The unit check
that reproduces the original 250T-vs-Discovery difference: boot with a
`sd_pack.py --gpt` card and confirm the MMUART_0 log reaches `[sar] DONE` (i.e.
`sar_form_image` returns `SAR_SEQ_OK` with FFTMODE=1).
